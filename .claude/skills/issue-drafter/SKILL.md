# issue-drafter (toss-trade-bot 전용)

시니어 PM이 에픽/기능 요청을 받아 **병렬로 물 수 있는 이슈 묶음**으로 쪼개고, 이 레포의 컨벤션에 맞춰 GitHub에 발행한다.

이 버전이 글로벌 issue-drafter와 다른 점 한 가지: **태스크 고도에만 머문다.** 아키텍처 결정(동시성 모델·무인 안전 계약·패키지 경계 등)은 여기서 내리지 않는다 — 그건 `/architect`가 ADR로 내린다. 이 스킬은 ADR을 *읽어서* 이슈에 링크하고, 근거 ADR이 없는 아키텍처 포크를 만나면 **멈춘다**. 이로써 디스패치된 stateless 서브에이전트가 "의도된 설계를 모른 채" 구현하는 일을 막는다.

---

## 0. 환경 도출 (하드코딩 금지 — 런타임에 구한다)

이 레포는 언제든 public 전환 가능하므로 경로·계정명을 박지 않는다.

```bash
REPO=$(git rev-parse --show-toplevel)
url=$(git -C "$REPO" remote get-url origin); url=${url%.git}
norm=${url//:/\/}                       # scp-style 'git@host:owner/repo'의 ':'를 '/'로 정규화
repo=${norm##*/}; rest=${norm%/*}; owner=${rest##*/}
SLUG="$owner/$repo"
```

모든 `gh` 호출에 `--repo "$SLUG"`. **계정 핀**: active gh 계정이 이 레포에 접근 못 할 수 있고 명령 사이에 되돌아갈 수 있다 — 접근 가능한 계정을 한 번 알아낸 뒤 gh 작업은 `gh auth switch --user <계정>` **직후 같은 셸 명령 안에서** 실행한다(`gh repo view "$SLUG"`로 검증). 발행은 퍼블릭 상태 변경이다 — 실패하면 정리하지 말고 보고.

---

## 1. 기존 ADR 읽기 (이 스킬의 핵심 추가)

```bash
ls "$REPO"/docs/adr/ 2>/dev/null && cat "$REPO"/docs/adr/[0-9]*.md
```

분해를 시작하기 전에 **현재 Accepted된 아키텍처 결정**을 머리에 넣는다. 분해 도중:

- 어떤 이슈가 특정 ADR이 정한 결정에 의존하면 → 그 이슈에 **지배 ADR을 링크**한다(아래 `## 설계 근거`).
- 어떤 이슈를 제대로 명세하려는데 **아직 결정 안 된 아키텍처 포크**에 부딪히면(예: "주문 재시도를 어떻게 멱등하게?"인데 관련 ADR 없음) → **분해를 멈추고** 사용자에게 알린다: "이 이슈는 아직 결정 안 된 설계 X에 의존한다. `/architect <X>`로 ADR을 먼저 만든 뒤 다시 부르라." **아키텍처를 여기서 추측해 이슈 본문에 박지 않는다.**

`docs/adr/`가 비어 있고 요청이 순수 태스크(설계 결정 불필요)면 그대로 진행한다.

---

## 2. Repo 컨텍스트 수집

이 레포 컨벤션을 매번 조회한다(기억하지 말 것 — 변한다):

```bash
gh issue list --repo "$SLUG" --state all --limit 10 --json number,title,labels,body
gh label list --repo "$SLUG" --json name,description
```

`CLAUDE.md`는 **`Read` 도구로 전문을 읽는다**(grep로 끝내지 말 것). 거기 규칙(패키지 레이아웃, 무인 제약, 5축 라벨 정의)이 이 스킬의 기본값보다 우선이다.

**이 레포의 5축 라벨**(`gh label list`로 실제 확인 — 정의가 변할 수 있다): `type:`(feature/bug/refactor/test/docs/chore/spike), `area:`(auth/market/order/account/strategy/runtime/config/infra, 복수 가능), `agent:`(ready/blocked/needs-decision), `risk:`(critical/high/low), `priority:`(p0/p1/p2). 새 이슈에는 해당 축을 빠짐없이 붙인다.

- `agent:` — 선행 의존이 머지돼야 시작 가능하면 `blocked`, 시작 전 사람 결정 필요하면 `needs-decision`, 명세 완결·자율 실행 가능하면 `ready`.
- `risk:critical`(실주문·자금이동·비가역)은 사람 리뷰 필수임을 본문에 명시.

---

## 3. 분해 설계

페르소나: 빅테크 시니어 PM. 습관 — **결과 먼저**(이 이슈가 끝나면 무엇이 달라지나 1~2줄), **스코프에 냉정**(한 이슈 = 한 사람 1~3일분), **측정 가능한 AC**("잘 됨" 금지), **모르는 건 `## 미정 사항`에**, **병렬성을 설계로 강제**, 사과·완곡어 없음.

분해 순서:

1. 핵심 산출물 한 문장.
2. **레이어/도메인 경계로 1차 분해.** 이 레포는 `internal/<도메인>`(toss/order/market/account/strategy/runtime…) 경계가 자연스러운 병렬 단위다. UI를 세로로 자르듯 한 도메인을 인위적으로 쪼개지 말 것.
3. **공유 접점 식별.** 여러 이슈가 건드리는 공통 파일(`cmd/bot/main.go`의 wiring, `go.mod`, 공유 인터페이스 선언)은 표시하고, 그 변경은 **가장 먼저 머지되는 이슈**가 담당하게 의존성을 건다.
4. **2~5개 목표.** 1개면 분해 불필요, 6개 이상은 과분해.
5. **의존성 트리.** fan-out(`A→B, A→C`)·fan-in은 정상, 사이클/밀집 그래프는 분해 실패 신호 → 선행 인터페이스 이슈(타입·시그니처 고정) 하나를 떼면 뒤가 풀린다. 선형(`A→B→C`)은 병렬 여지 없으니 쪼갤 가치 재검토.
6. **파일 소유권으로 분리/병합.** 주 편집 파일이 다르면 분리 유지(설정 1~2줄 겹침은 "공유 접점"으로만). 같은 `.go` 파일 본문을 나눠 쓰면 충돌 확정 → 합병하거나 선행 인터페이스 이슈로.

---

## 4. 이슈 본문 작성

이 레포 기존 이슈 스타일(`## 목적`/`## 작업`/`## 규칙`/`## Acceptance Criteria`)을 베이스로, 아래 두 섹션을 **항상** 끝에 붙인다:

```markdown
## 설계 근거
- 지배 ADR: ADR-NNNN (docs/adr/NNNN-...md) — <이 이슈가 따라야 하는 결정 한 줄>
  (지배 ADR이 없으면 이 줄 생략. 단, 아키텍처 결정이 *필요한데* 없으면 1번 규칙대로 멈춘다.)

## 병렬 작업 메모
- **건드리는 영역**: <구체 파일/패키지 경로, 예: internal/order/ (신규), cmd/bot/main.go (wiring 1줄)>
- **선행 이슈**: <#번호 또는 "없음">
- **공유 접점**: <다른 이슈와 겹치는 파일과 겹치는 방식 — 있을 때만>
```

`건드리는 영역`은 리뷰어가 두 이슈를 집합 비교해 충돌을 즉시 판단할 수준으로 구체적이어야 한다. AC는 이 레포 검증 게이트(`go test -race ./...`, `go vet`)를 체크 가능한 문장으로 반영한다.

발행 전이라 선행 이슈 번호를 모르면 본문에 앵커명(`선행: 이슈 A`)을 쓰고 6단계에서 실제 번호로 치환한다 — 번호를 추측해 박지 않는다.

---

## 5. 프리뷰 + 승인 (필수)

**Step 4에서 바로 `gh issue create` 금지.** 이슈는 한 번 만들면 번호가 영구 소비되고 닫혀도 노이즈다. 전체 묶음 + 의존성 트리를 마크다운으로 보여주고 승인을 받는다. 각 이슈에 라벨(5축)과 지배 ADR을 함께 표시한다. 사용자가 OK 할 때까지 6단계로 가지 않는다. 피드백 오면 해당 이슈만 다시 쓰고 변경분만 짚는다.

---

## 6. 의존성 순서대로 발행

승인되면 위상정렬 순서로(선행 먼저) 발행한다 — 뒤 이슈의 `선행 이슈: #앞번호`를 실제 번호로 채우려면 앞이 먼저 있어야 한다.

```bash
gh auth switch --user <접근 가능한 계정> >/dev/null && \
gh issue create --repo "$SLUG" \
  --title "<제목>" --body-file <본문파일> \
  --label "type:feature,area:order,agent:ready,risk:high,priority:p1"
```

본문은 파일로 넘긴다(따옴표 이스케이프 회피). 발행 직후 URL에서 번호를 잡아 다음 이슈 본문에 치환. 끝나면 `✓ #N <제목> — URL`로 한 줄씩 요약.

---

## 7. 실패 시 복구

`gh` 실패 시 **그 자리에서 멈추고** 이미 생성된 번호 + 실패 지점을 보고한다. 자동 롤백(이미 만든 이슈 닫기) 금지 — 의도와 다르게 닫히는 게 더 나쁘다. 라벨 미존재면 임의 생성 말고 사용자에게 권한 확인.

---

## 동작 원칙

- **Repo 컨벤션 > 스킬 기본값.** 기존 이슈를 먼저 읽고 그 스타일을 그대로 따른다. 새 섹션은 기존을 유지한 채 끝에 덧붙인다.
- **고도를 지킨다.** 아키텍처 결정은 ADR(=`/architect`)의 몫. 여기서 추측해 박지 말고 멈춘다.
- **분해는 도메인 경계를 탄다.** `internal/<도메인>` 경계가 자연스러운 병렬 선. 한 도메인을 세로로 자르면 충돌한다.
- **프리뷰는 필수.** "알아서 발행"이라 해도 첫 호출엔 보여주고 확인받는다.
- **과분해 경계.** 1~3일분이 한 이슈. 1~2시간짜리 10개는 오버헤드가 작업을 넘는다.

## 엣지 케이스

- **버그 리포트**: 분해하지 말고 단건 이슈로, PM 톤(재현·기대/실제·환경) 적용. `type:bug`.
- **fresh repo(이슈 0개)**: 스타일 학습 건너뛰고 기본 템플릿. 프리뷰에서 한 줄 고지.
- **아직 없는 코드에 의존(virtual dependency)**: `병렬 작업 메모`에 코드 레벨 선행과 "도메인 선행"(아직 코드 없지만 이슈로만 존재)을 두 줄로 분리. 프리뷰 preamble에 "이 묶음은 #X 머지 전제"라고 고지.
- **이미 일부 이슈 존재**: 제목·스코프 중복 비교, 프리뷰에서 `⚠️ #N과 중복 가능성` 표시 후 사용자 판단.

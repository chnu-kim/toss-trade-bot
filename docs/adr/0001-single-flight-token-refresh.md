# ADR-0001: OAuth2 토큰은 단일 매니저에서 single-flight로 발급·갱신한다

- **Status**: Accepted
- **Date**: 2026-06-29
- **Deciders**: chnu-kim
- **관련 이슈/PR**: #1, PR #5

## Context

Toss Open API 인증은 OAuth2 **Client Credentials Grant** 전용이다(`POST /oauth2/token`). 토큰 수명은 86400초(24시간)이고 **refresh token이 없다**.

결정적 제약: **클라이언트당 유효 토큰은 1개뿐이다.** 토큰을 새로 발급하면 기존 토큰이 즉시 무효화된다. 따라서 불필요한 재발급은 현재 살아 있는 토큰을 죽인다.

이 봇은 24시간 무인 실행되며 여러 goroutine이 동시에 API를 호출한다. 만료 시점 근처에서 다수의 goroutine이 각자 "토큰 만료됐네" 하고 동시에 재발급을 요청하면, 서로가 서로의 토큰을 무효화해 인증 폭주(thundering herd → 연쇄 401)가 발생한다. 무인 안전성(CLAUDE.md "무인 실행 제약")의 1순위 위협이다.

## Decision

토큰의 발급·캐시·갱신을 **단일 토큰 매니저 한 곳**에 집중한다. 매니저는:

- 토큰을 메모리에 캐시하고 만료 전 여유를 두고 선제 갱신한다.
- 동시에 들어온 다수의 갱신 요청을 **single-flight**로 합친다 — 실제 발급은 1회만 일어나고, 나머지 호출자는 그 결과를 공유받는다.
- 모든 요청에 `Authorization: Bearer`를 주입한다.
- 401 응답 시 토큰을 강제 만료시키고 갱신한 뒤 **조회 요청에 한해 1회** 재시도한다(쓰기/주문은 재시도 금지 — 별도 결정).

## Alternatives considered

- **요청별 mutex로 갱신을 직렬화** — 기각: 갱신 코드 자체는 보호되지만, "만료를 감지한 N개의 goroutine이 mutex를 차례로 잡고 각각 한 번씩 재발급"하는 중복 발급을 막지 못한다. 토큰 1개 제약 하에서는 이 중복이 곧 상호 무효화다. single-flight는 "이미 진행 중인 발급이 있으면 새로 시작하지 않고 그 결과를 기다린다"는 점에서 mutex와 다르다.
- **토큰을 소유한 전용 goroutine + 채널(actor 패턴)** — 기각: 단일 토큰·단일 매니저 규모에 비해 과설계다. 무인 운영에서 중시하는 단순함·재시작 단순성을 해친다.
- **매 호출마다 새 토큰 발급** — 기각: 토큰 1개 제약을 정면으로 위반하고 레이트 리밋을 소진한다.

## Consequences

- (좋음) 토큰 접근 경로가 하나로 모여 `go test -race`로 동시성 정확성을 검증할 수 있다(이슈 #1 Acceptance Criteria).
- (비용) single-flight 동기화 로직과 그 동시성 테스트가 반드시 필요하다.
- (제약 전파) 프로세스가 **여러 개** 뜨면 각자의 매니저가 서로 토큰을 무효화한다 → **토큰 발급은 단일 프로세스에서만** 해야 한다는 운영 제약이 따라온다(후속 ADR 후보: 멀티 인스턴스 시 토큰 발급 책임 분리).
- (재시작) 프로세스 재시작 시 캐시는 휘발하고 기동 후 첫 호출에서 재발급한다. 토큰 1개 제약과 호환된다(직전 토큰을 무효화하지만, 죽은 프로세스의 토큰은 어차피 쓰이지 않는다).

# ADR-0004: 킬 스위치는 신규 노출 제출 엣지 하나에 걸리는 fail-closed 가드이고, 전역 halt만 영속한다

- **Status**: Accepted
- **Date**: 2026-07-02
- **Deciders**: chnu-kim
- **관련 이슈/PR**: (TBD)

## Context

이 봇은 24시간 무인 실행되고 주문 제출은 **비가역(돈이 움직임)**이다. CLAUDE.md "무인 실행 제약"은 킬 스위치를 명시적으로 요구한다: **비정상(연속 실패, 예상외 손실, 토큰 갱신 불가) 시 신규 주문을 멈추는 안전장치.** 그리고 ADR-0003이 이 결정을 후속으로 넘겼다 — *"킬 스위치 메커니즘(국소·전역 차단 상태, 빈발 임계, 해제 절차, 알림 채널)은 별도 ADR"*.

이 ADR은 답한다: **킬 스위치는 정확히 어느 행위를 막고 무엇은 막지 않는가, 스스로 고장 나면 어느 쪽으로 실패하는가, 재시작을 어떻게 견디는가, 무엇이 그것을 켜고 어떻게 푸는가.**

결정을 가르는 힘:

- **비가역성.** 돈이 나가는 행위(신규 주문)만 위험하다. 진실을 확인하는 행위(reconciler·조회·토큰 갱신)는 아무리 돌아도 위험하지 않고, 오히려 위급할수록 더 돌아야 한다.
- **무인 안전 불변식(CLAUDE.md).** 죽지 않는다 · 재시작 안전 · 로컬 상태 불신(SoT는 API) · 조회는 백오프 재시도 OK, 주문 재전송은 자동 금지.
- **단일 프로세스(ADR-0001).** 토큰 1개 제약이 토큰 발급을 단일 프로세스로 확정했다 → 킬 스위치는 in-process 공유 상태 + durable 백킹이면 충분하고, 분산 조율은 범위 밖이다.
- **journal + reconciler(ADR-0002/0003).** intent는 write-ahead journal(outbox)에 남고, orderId 상세조회로 진실을 얻으며, orderId 없는 ambiguous submit은 fail-closed(국소 차단, 빈발 시 전역 킬 스위치로 에스컬레이션)한다.
- **API 제약(ADR-0003 실측).** `holdings`는 집계 포지션만, `trades`엔 orderId가 없다 → 개별 주문 단위 손익(P&L)을 깔끔하게 감지할 수 없다.
- **영속 저장소.** SQLite/Postgres를 쓸 예정이나 확정은 **별도 ADR**로 다룬다(ADR-0002의 "append-only 파일 + fsync" 표현 조정 포함). 이 ADR은 "journal과 **동일한** 트랜잭션 저장소"라고만 적어 저장소 확정에 독립적이게 한다.

## Decision

1. **킬 스위치는 신규 노출 제출 엣지 하나에 걸리는 동기 술어다.** order 계층이 신규 노출 주문을 제출하기 직전(ADR-0002의 `prepared` journal append 이전)에 `CanSubmit(symbol) → (allowed, reason)`을 묻고, `allowed=false`면 제출을 시작조차 하지 않는다. 결정적으로 **무엇에 안 걸리는지**를 명시한다 — halt 중에도 계속 돈다:
   - **reconciler** (ADR-0003 point 4: query-only, 절대 재제출 안 함 → 안전. halt 중에도 in-flight 주문의 진실을 계속 확정해야 한다).
   - **토큰 매니저** (ADR-0001: 죽이면 복구 자체가 불가).
   - **모든 조회**(holdings/orders/prices/orderbook).

   **check와 실제 제출은 원자적이어야 한다(TOCTOU 봉쇄).** `CanSubmit`은 다른 goroutine이므로, 이 프로세스의 어떤 컴포넌트든 `CanSubmit` 통과 후·비가역 API 제출 전 사이에 halt를 걸 수 있다 — 그 창으로 신규 노출이 샌다. 따라서 **비가역 제출 직전(`submit-attempted` 마커 append 직전)에 fail-closed 최종 재확인**을 둔다: 그 시점 halt가 걸려 있으면(또는 첫 `CanSubmit`에서 잡은 kill-switch generation이 바뀌었으면) 제출을 **중단**한다. 중단된 intent는 `prepared`만 있고 `submit-attempted`가 없으므로, reconciler가 ADR-0003의 `aborted-before-submit` terminal 경로로 깨끗이 닫는다 — 새 ambiguity를 만들지 않는다. (generation/reservation vs 최종 재확인 중 구현 선택은 구현 이슈, ADR은 "제출 임계구역은 race-safe·fail-closed" 계약만 고정한다.)

2. **가드는 "신규 노출"만 막고 "정리(청산·취소)"는 막지 않되, 정리 여부는 저신뢰 경계에서 독립 검증한다.** 킬 스위치의 목적은 새 위험을 안 키우는 것이지 이미 진 위험을 정리하는 행위(손절 매도·미체결 취소)를 봉쇄하는 게 아니다 — 위급할 때 정리마저 막으면 손해가 커진다. 전략이 각 주문에 "신규 노출 / 정리" 의도를 태그로 부여하되(주문의 의미는 전략이 안다 — ADR-0002 경계), **그 태그는 신뢰 대상이 아니다.** 위험을 만드는 컴포넌트(전략)가 자기 태그로 안전장치를 우회하게 두면, 전략 버그·부호 오류·stale 포지션 뷰·재시도 경로가 신규 노출을 "정리"로 오분류해 halt 중에 통과시킨다.

   따라서 **order/risk 계층이 정리 태그를 독립 검증**한다: 해당 주문이 **현재 포지션(holdings) + 미해결 open-intent 상태에 비추어 노출을 실제로 줄이는지** 확인하고, **줄임을 증명하지 못하면 신규 노출로 취급(fail-closed)** 해 킬 스위치의 차단 대상에 넣는다. 즉 우회는 전략의 주장이 아니라 order/risk의 검증을 통과해야만 허용된다. (검증 알고리즘·포지션 조회 방식은 구현 이슈. 이 검증 때문에 order/risk → account(포지션) 의존이 생긴다.)

3. **가드 자신은 fail-closed다 — 기동 replay 완료 전까지 포함.** 상태를 못 읽거나, 체크가 에러나거나, 기동 시 저장소에서 halt 상태 로드가 실패하면 = **blocked로 취급**한다. 안전장치가 애매할 때 통과시키면(fail-open) 정작 위험한 순간 무력하다. **여기에 기동 창을 명시적으로 포함한다**: 종목별 차단은 persist하지 않고 reconciler 재도출에 의존하는데(point 4), 재도출은 즉시가 아니라 **journal 미해결 intent 스캔이 끝나야** 완료된다. 그 사이 `CanSubmit`이 빈 미러를 보고 통과시키면 재시작이 곧 종목별 보호 우회다. 따라서 **기동 시 신규 노출 제출은 미해결 intent 스캔·재-차단이 끝날 때까지 fail-closed(전면 blocked)** 로 두고, 스캔 완료 후에야 제출을 연다. (조회·reconciler·토큰 갱신은 기동 즉시 돈다 — 막는 건 신규 노출뿐.)

4. **두 scope: 종목별(per-symbol) 차단과 전역(global) halt.** 재시작 내구성은 **비대칭**이며, 규칙은 *"재시작 시 journal에서 deterministic하게 재도출되는 halt는 persist하지 않고, 재도출되지 않는 halt는 반드시 persist한다"*:
   - **종목별 차단 = persist 안 함.** ambiguous submit(ADR-0003)발 종목 차단은 journal에 `unresolved-ambiguous` intent가 남아 있으므로, 재시작 시 reconciler가 journal 미해결 intent를 읽고 **재-차단**한다. 별도 영속은 상태 중복 → 어긋남 위험만 만든다. **단, 재도출은 즉시가 아니므로 그 사이 창은 point 3의 기동 replay-gate(스캔 완료 전 제출 전면 fail-closed)로 닫는다** — 이 gate가 있어야 "persist 안 함"이 안전하다.
   - **전역 halt = persist.** 연속 실패·빈발 임계발 전역 halt는 카운터가 휘발한다. 메모리에만 두면 재시작 시 카운터가 0으로 초기화되어 **임계를 다시 채우는 동안 주문이 유출**된다(재시작 = 안전장치 우회). 따라서 전역 halt 상태+사유를 **journal과 동일한 트랜잭션 저장소에 영속**하고, 재시작 시 그 상태를 읽어 **halted로 기동**한다. **두 번째 영속 계층을 발명하지 않는다.**

5. **상태 모델: 저장소 = 영속 진실, 메모리 = 값싼 읽기 미러.** `CanSubmit`은 모든 주문마다 호출되는 뜨거운 경로이므로 매번 저장소를 때리지 않는다. trip/clear 시 저장소에 쓰고 in-process 미러를 갱신하며, 기동 시 저장소에서 로드해 미러를 채운다. ADR은 이 *성질*("값싼 in-process 읽기 + 동일 저장소 영속 백킹")만 고정하고, 구체 sync primitive(`atomic.Bool`/`atomic.Value` vs `RWMutex` 등)는 **구현 이슈에 맡긴다** — atomic 계열이 lock-free 읽기에 유력한 후보다.

6. **해제(reset) 절차.** *"자동으로 푼다는 건 = 그 조건에서 자동 재개해도 안전하다고 결정했다는 뜻이다."*
   - **종목별 = 조건 해소 시 자동 clear.** reconciler가 애매함을 해소하면(예: orderId를 확정 상태로 마감) 더 이상 위험하지 않으므로 그 종목 차단을 자동 해제한다.
   - **전역 halt = 사람 수동 해제만.** 시스템 이상은 근본 원인이 안 고쳐진 채 봇이 스스로 재개하면 또 사고를 낸다. 명시적 사람 해제(clear) 조작이 있어야만 재개한다.
   - **수동 해제 UI(CLI/관리 API/직접 DB 조작 등)는 구현 이슈로 미룬다** — ADR은 "명시적 해제 조작 없이는 재개 없음"이라는 계약만 박는다.

7. **트리거 API는 제네릭하게, 임계 판단은 한 곳에.** 킬 스위치는 상태와 `Trip(scope, reason)`/reset API를 소유하는 sink이고, 위험 신호를 세는 임계 카운터는 **킬 스위치 한 곳**에 둔다(신호 발신처는 판단하지 않는다 → 테스트 표면이 하나). `Trip(scope, reason)`은 어떤 사유든 받도록 제네릭하게 두되, **지금 구현 가능한 트리거만** 명시한다:
   - ambiguous submit 1건 → **종목별** 차단 (ADR-0003에서 확정).
   - ambiguous submit **빈발**(연속/단위시간당 임계 초과) → **전역** halt (ADR-0003이 이 ADR에 위임한 임계 정의).
   - 주문 **연속 실패** 임계 초과 → **전역** halt.
   - **토큰 갱신 불가** → **전역** halt.
   - **"예상외 손실"은 미래 트리거 후보로만 남긴다** — API가 못 받친다. `holdings`는 집계뿐이고 `trades`엔 orderId가 없어 손익을 개별 주문에 엮을 수 없다(ADR-0003). 체결/P&L 추적 설계가 생길 때 별도로 붙인다.

   **에스컬레이션 임계 상태(카운터·시간 윈도우)에도 point 4의 재시작 비대칭 규칙을 그대로 적용한다.** 트립된 halt만 지키고 트립 직전까지 쌓인 카운터가 휘발하면, 크래시/재시작 루프가 임계 도달 전에 카운터를 계속 리셋해 에스컬레이션 자체를 우회한다(무인 24/7 + 의존성 열화 시 크래시 루프는 현실적 위협). 따라서:
   - **journal에서 재구성 가능한 신호는 기동 시 재계산한다** — ambiguous 빈도는 `unresolved-ambiguous` intent와 그 타임스탬프(ADR-0002 `submit-attempted` 시각)에서 다시 센다. 별도 persist 없음(공짜).
   - **journal에서 재구성 불가한 신호는 카운터+윈도우를 동일 트랜잭션 저장소에 persist한다** — 토큰 갱신 실패처럼 순수 런타임 신호가 여기 해당한다(희소 신호라 쓰기 부담 작음).
   - **기동 시 임계 상태를 복구(재계산/로드)하지 못하면 fail-closed** — 복구 실패를 "증거 없음"이 아니라 blocked로 취급한다.

8. **패키지·알림.** 킬 스위치는 **`internal/killswitch`** leaf 패키지에 둔다 — order가 이걸 import하고 컴포넌트가 신호를 보내며, killswitch는 order/strategy를 import하지 않는다(순환 금지). 알림은 **notifier 인터페이스(seam)만** 정의하고("halt가 켜지면 notifier를 호출한다"), 구체 채널(슬랙/이메일/텔레그램 등)은 out of scope다. (형제 안전장치 — market-hours 게이트·backoff 상태 등 — 이 실제로 여럿 생기면 그때 `internal/safety/killswitch`로 그룹핑을 재고한다. 지금은 외동 네스팅을 피해 평평하게 둔다.)

## Alternatives considered

- **모든 주문 차단(청산·취소 포함)** — 기각: 위급 시 손절 매도·포지션 정리마저 막아 손해를 키운다. 킬 스위치 목적은 신규 노출 억제이지 리스크 축소 행위 봉쇄가 아니다.
- **order가 신규/정리를 스스로 추측** — 기각: 주문의 의미(신규 노출인가 정리인가)는 전략이 안다(ADR-0002 경계). 의미 모르는 계층이 판단하면 오분류 → 킬 스위치 우회.
- **가드 fail-open(못 읽으면 통과)** — 기각: 안전장치가 애매할 때 통과시키면 정작 위험한 순간 안전장치가 고장 나 있어도 주문이 나간다 — 안 쓰느니만 못하다.
- **전역 halt를 메모리에만 보관(persist 안 함)** — 기각: 재시작이 곧 안전장치 우회다. 카운터 초기화 → 임계 재충전 중 주문 유출. 비가역 자금에서 수용 불가.
- **종목별 차단도 영속** — 기각: 불필요하다. journal의 `unresolved-ambiguous` intent에서 reconciler가 재시작 시 공짜로 재도출한다. 중복 상태는 어긋남만 만든다.
- **halt용 두 번째 영속 계층 신설** — 기각: journal과 같은 저장소를 재사용한다 — 단일 진실, journal 기록과 halt 기록을 한 트랜잭션으로 묶어 "주문은 실패 기록됐는데 halt는 안 켜진" 어긋남을 차단한다.
- **전역 halt 자동 재개(조용해지면 스스로 재시작)** — 기각: 근본 원인(네트워크·API 장애·버그) 미해결 채 재발한다. 시스템 이상은 사람 확인이 안전 조건이다.
- **임계 판단을 신호 발신처(order/reconciler)에 분산** — 기각: 위험 로직이 흩어져 테스트 표면이 여러 곳이 된다. 카운팅·임계는 killswitch 한 곳에 모은다.
- **"예상외 손실" 트리거를 지금 포함** — 기각: `holdings` 집계·`trades` orderId 부재로 P&L을 주문에 엮을 수 없다. API가 못 받치는 트리거를 ADR이 약속하면 거짓 계약이다. 후보로만 남긴다.
- **`internal/safety/killswitch` 네스팅** — 기각(현시점): 형제 안전장치가 아직 없어 외동 네스팅 = 과설계다. 형제가 생기면 그때 이동한다.
- **분산 킬 스위치 / 멀티 인스턴스 조율** — 범위 밖: ADR-0001이 토큰 1개 제약으로 단일 프로세스를 확정했다. in-process 공유 상태 + durable 백킹이면 족하다.
- **특정 sync primitive(atomic/mutex)를 ADR에 못박기** — 기각: 구현 자유를 뺏는다. ADR은 "값싼 in-process 읽기" 성질만 고정하고 primitive는 구현 이슈에 맡긴다.
- **`CanSubmit` 단일 체크만(제출 직전 재확인 없음)** — 기각(codex adversarial-review 지적): check와 비가역 제출이 원자적이지 않으면 그 사이에 다른 컴포넌트가 halt를 걸어도 신규 노출이 샌다(TOCTOU). 제출 임계구역에 fail-closed 최종 재확인(또는 generation/reservation)을 둔다.
- **전략의 "정리" 태그를 검증 없이 신뢰** — 기각(codex adversarial-review 지적): 위험을 만드는 컴포넌트가 자기 주장으로 안전장치를 우회한다. 전략 버그·부호 오류·stale 포지션 뷰가 신규 노출을 정리로 오분류해 halt 중 통과시킨다. order/risk가 포지션·open-intent로 독립 검증하고, 증명 실패 시 신규 노출로 취급(fail-closed)한다.
- **기동 즉시 제출 허용(replay 완료를 안 기다림)** — 기각(codex review P1 지적): 종목별 차단을 persist 안 하는 대신 reconciler 재도출에 의존하는데, 재도출 완료 전 창에서 `CanSubmit`이 빈 미러를 보고 통과시켜 재시작이 종목별 보호 우회가 된다. 기동 시 신규 노출 제출을 스캔 완료까지 fail-closed로 둔다.
- **트립된 halt만 영속, 임계 카운터는 휘발** — 기각(codex adversarial-review 2차 지적): 임계 도달 전 재시작이 연속 실패·ambiguous·토큰 갱신 실패 카운터를 리셋해 에스컬레이션을 우회한다(크래시 루프에서 특히 위험). 카운터에도 재시작 비대칭 규칙을 적용한다 — 재구성 가능 신호는 재계산, 불가 신호는 persist, 복구 실패는 fail-closed.
- **모든 임계 카운터를 무조건 persist(뜨거운 경로에 durable 증가)** — 기각: ambiguous 빈도 등 journal에서 공짜로 재계산되는 신호까지 매번 durable write하면 불필요한 쓰기 부담·상태 중복이 생긴다. "재구성 불가한 신호만 persist"가 비용/안전 균형점이다.
- **재시작 시 미해결 안전 신호 있으면 무조건 fail-closed + 사람 검토(강수)** — 기각: 가장 보수적이나, 크래시 루프에서 재시작마다 사람 개입을 요구해 무인성을 과도하게 희생한다. 재계산으로 증거를 복원할 수 있으면 자동 복구가 비례적이고, 복원 불가일 때만 fail-closed면 충분하다.

## Consequences

- (좋음) 비가역 신규 노출만 차단하고, 진실 확인 경로(reconciler·조회·토큰 갱신)는 halt 중에도 살아 있어 재시작 복구와 in-flight 추적이 끊기지 않는다.
- (좋음) 전역 halt를 영속하고 halted로 기동하므로 **재시작 = 안전장치 우회 윈도우**가 제거된다.
- (좋음) 종목별 차단은 journal에서 재도출되어 상태 중복·어긋남이 없다.
- (좋음) 임계 카운팅·trip 로직이 killswitch 한 곳에 모여 `go test -race`·단위 테스트로 검증할 표면이 하나다(시계·신호 mock 주입).
- (좋음) notifier seam 덕에 알림 채널이 미정이어도 진행 가능하고, 채널이 정해져도 killswitch는 안 바뀐다.
- (좋음) fail-closed 구멍 4개(TOCTOU 제출 레이스·기동 replay 창·전략 태그 신뢰·휘발 임계 카운터)를 닫아, "신규 노출은 halt/애매 상황에서 절대 못 샌다"가 전략 신뢰·타이밍·재시작과 무관하게 성립한다. 특히 크래시/재시작 루프가 에스컬레이션 진행을 지우지 못한다.
- (비용) 전략은 각 주문에 "신규 노출 / 정리" 의도 태그를 부여하되 그 태그는 **신뢰되지 않는다** — order/risk가 포지션·open-intent로 독립 검증한다. 이로 인해 **order/risk → account(포지션) 의존**과 검증 로직이 생기고, 증명 못 하는 정리는 신규 노출로 강등되어 halt 중 막힌다(보수적 오차).
- (비용) 제출 임계구역에 fail-closed 최종 재확인(또는 generation/reservation)이 필요하고, **기동 시 미해결 intent 스캔·임계 상태 복구 완료 전까지 신규 노출 제출이 지연**된다(조회·reconciler·토큰은 즉시 가동).
- (비용) 재구성 불가 신호(토큰 갱신 실패 등)의 임계 카운터+윈도우를 저장소에 persist하고 기동 시 복구해야 한다. 재구성 가능 신호(ambiguous 빈도)는 기동 스캔에서 재계산으로 흡수된다.
- (비용) 뜨거운 경로 in-process 미러 + 저장소 영속의 이중화, 기동 시 로드, 로드 실패 시 fail-closed 처리가 필요하다.
- (제약 전파) 전역 halt는 사람 수동 해제를 요구한다 → 무인성에 사람 개입점이 생긴다. 이 개입 빈도(=실제 시스템 이상 빈도)가 운영 지표가 된다.
- (계약 전파) killswitch는 저장소가 확정되면(별도 ADR) journal과 동일한 트랜잭션 경계를 공유한다.
- (후속)
  - **영속 저장소 결정 ADR** — SQLite/Postgres 확정 및 ADR-0002의 "append-only 파일 + fsync" 표현 조정.
  - **알림 채널 구체화** — notifier 구현.
  - **"예상외 손실" 트리거** — 체결/P&L 추적 설계 선행 후 별도.
  - **빈발 임계·settle window 파라미터·연속 실패 임계·수동 해제 UI** — 구현 이슈에서 단위 테스트로 경계 커버. **"임계 도달 전 재시작"(카운터 재계산/persist 복구) 시나리오를 반드시 테스트에 포함**한다.
  - **정리(reducing) 주문 검증 알고리즘** — 포지션·open-intent 대비 노출 감소 증명 방식, order/risk↔account 경계 — 구현 이슈.
